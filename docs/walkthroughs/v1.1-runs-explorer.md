## 🚀 Key Features

### 1. Runs Explorer Dashboard

A first-class evidence browser integrated into the Operator Console, providing a real-time, normalized view of all experiments in the `runs/` directory.

- **Status Badges**: Visual indicators for Running, Queued, Completed, Failed, and Interrupted states.
- **Artifact Indicators**: Instant visibility into summaries and log availability.
- **Deterministic Status Resolution**: Backend status derived via strict precedence logic.

---

### 2. Forensic Detail Inspector

Operators can drill down into any run and inspect its core artifacts side-by-side:

- **Manifest** → Immutable launch intent (contract)
- **Runtime State** → Live lifecycle (PID, timestamps)
- **Summary** → Terminal evidence (metrics, errors)

This enables immediate validation of **intent vs execution vs outcome**.

---

### 3. ⚙️ Execution Integrity Improvements

- **Merge-Safe Status Writes**  
  `StateManager.write_status` preserves historical fields (e.g., `last_completed_exp_id`) across updates.

- **Worker Heartbeat Consistency**  
  Queue depth is propagated through all state transitions, ensuring UI accuracy.

- **Correct Failure Semantics**
  - Failed runs no longer update `last_completed_exp_id`
  - Terminal success and failure are strictly separated

## 🧠 Deterministic Precedence Logic

Status is derived from filesystem artifacts using strict, governance-safe ordering:

| Priority | Status | Trigger |
| :--- | :--- | :--- |
| 1 (High) | `failed` | `run_summary.json` exists + `status=="failed"` |
| 2 | `completed` | `run_summary.json` exists |
| 3 | `running` | `runtime_state.json.status == "running"` |
| 4 | `interrupted` | `runtime_state.json.status == "interrupted"` |
| 5 | `queued` | `runtime_state.json.status == "queued"` |
| 6 (Low) | `unknown` | No artifacts found |

This transforms:
> implicit filesystem ambiguity → deterministic state machine
